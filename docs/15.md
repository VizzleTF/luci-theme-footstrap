# Бенчмарк навигации: footstrap (SPA) vs bootstrap (полная перезагрузка)

Замеряет, насколько быстрее `luci-theme-footstrap` с клиентским SPA-роутером
(см. [docs/14](14-spa-router.md)) открывает страницы по сравнению со штатной
`luci-theme-bootstrap`, при обходе стандартных страниц меню по очереди с полным
рендером каждой.

Скрипт: `bench/nav-benchmark.py`. Воспроизводимо на любом роутере 25.12+.

## Что и как меряется

Для каждой страницы — время в миллисекундах (wall-clock) от «намерения
перейти» до «view полностью отрисован»:

- **footstrap** — реальный клик по пункту меню → SPA-роутер переинстанцирует
  view в `#view`, без перезагрузки страницы;
- **bootstrap** — полный переход по URL → браузер перезагружает весь каркас,
  заново парсит/исполняет `luci.js`+`cbi.js`, до-тягивает переводы, пересобирает
  меню, затем рендерит view.

Обе темы рендерят контент в **один и тот же** `#view` (его эмитит диспетчер,
независимо от темы), поэтому условие «отрисовано» одинаковое и сравнение
честное — меряется воспринимаемое пользователем время «клик → страница готова».

«Отрисовано» = в `#view` есть дети, ничего не крутится, нет «Loading view…» —
и ни один узел не помечен как контент **предыдущей** страницы (см. «Две ловушки»).

### Метод (на каждую тему)

1. Активировать тему (`uci luci.main.mediaurlbase`), сбросить кэши LuCI.
2. Свежий контекст браузера, логин.
3. **Прогрев** — обойти все страницы один раз без замера (наполнить HTTP-кэш и
   кэш модулей LuCI → устойчивое состояние, «после того как всё прогрузилось»).
4. **Замеры** — `--runs` проходов (по умолчанию 5), берётся **медиана** по
   странице (глушит шум). Дополнительно считается число сетевых запросов за
   переход.

### Какие страницы

Определяются вживую из `/admin/menu`: **всё, что меню вообще может превратить в
ссылку** (`satisfied` + есть `title` — фильтр `ui.menu.getChildren()`) глубиной
от 3, чья **разрешённая** цель — стандартная страница LuCI. Сторонние приложения
исключены по построению: сверяется путь **view**, а не пункта меню, поэтому
приложение, повесившее себя под `admin/system`, в выборку не попадёт.

Было 14 страниц, стало **38**. Три ограничения старой выборки прятали 24 из них:

- **`len == 3`** — вкладки считались «не кликом по меню». Но их кликают, просто в
  `#tabmenu`, и SPA-роутер обслуживает их так же: Realtime (5), Logs (2),
  Administration (5), Firewall (5) — 17 страниц из 38.
- **`action.type == 'view'`** — мимо прошли `alias` (Firewall, System Log,
  Realtime Graphs), `firstchild` (Administration) и `template` (Overview). Это
  самые нажимаемые пункты меню: бенчмарк мерил тему везде, кроме её парадной двери.
- **префикс view-пути** — у `package-manager` его просто нет, и фильтр молча
  выкинул страницу.

Порядок обхода — древесный, и это не косметика: вкладка кликабельна только когда
её раздел открыт, а родительский пункт (`alias`/`firstchild`) в дереве всегда идёт
раньше своих детей — значит к моменту клика ссылка гарантированно есть в DOM.

Исключены намеренно, обе — по одной причине «время принадлежит не теме»:
`admin/status/channel_analysis` (5 с занимает аппаратное сканирование эфира, да и
гонять его 12 раз — глушить пользователю Wi-Fi) и `attendedsysupgrade` (ходит на
`sysupgrade.openwrt.org`, то есть померили бы интернет).

### Две ловушки в самом харнессе

Обе давали **правдоподобные, но ложные** числа — их стоит знать, прежде чем верить
любому такому бенчмарку.

1. **Спиннер как «навигация началась».** Старый `wait_render()` сперва ждал
   появления `.spinning` с таймаутом 3000 мс. Вьюха, чей модуль уже в кэше,
   рендерится **без единого кадра со спиннером** — ожидание отрабатывало весь
   таймаут, и в отчёт шло `~3017 ms`. Восемь страниц получили это число, и это
   были **самые быстрые** страницы: после починки — 9–63 мс.
2. **Старый контент как «страница готова».** SPA не очищает `#view`, пока новая
   вьюха не отрисуется, поэтому условие «в `#view` есть дети» истинно **сразу**
   после клика — оно бы засекло предыдущую страницу.

Оба лечатся одним приёмом: перед навигацией пометить детей `#view`
(`data-bench-old`), а готовностью считать «непомеченный контент, ничего не
крутится». LuCI рендерит вьюху через `dom.content(#view, …)`, то есть **заменяет**
детей — значит исчезновение меток и есть точный момент смены страницы. Спиннерный
гейт убран.

## Как повторить

```sh
# один раз: окружение (Playwright + Chromium)
python3 -m venv .venv
.venv/bin/pip install playwright
.venv/bin/python -m playwright install chromium

# запуск (тема автоматически возвращается к исходной в конце)
LUCI_PW=<пароль-root-роутера> .venv/bin/python bench/nav-benchmark.py \
    --ssh-host router --runs 3 --out bench/results-25.12.json
```

**Роутер должен быть подготовлен, иначе числа будут ложными.**

- **Все темы из `THEMES` обязаны быть установлены и зарегистрированы в `luci.themes`** —
  сейчас это `bootstrap`, `proton2025`, `footstrap`. Скрипт всего лишь пишет
  `luci.main.mediaurlbase`, и **никто этот путь не валидирует**: если шаблона темы нет,
  `runtime.uc` молча откатывается на первую рабочую тему из `luci.themes` (docs/01), и
  колонка «proton2025» честно замерит… `bootstrap`. Прогон не упадёт — он соврёт.
- **Ловушка №3 (после двух харнессных выше): приложение с инвазивным CSS ломает
  измерение.** Если на роутере стоит `luci-app-filemanager`, SSClash или другое
  приложение, инжектящее `<style>`, который красит штатные виджеты, footstrap
  **намеренно** уходит в полную загрузку на каждой навигации после захода на его
  страницу (гейт `documentPoisoned()`, [docs/14](14-spa-router.md)) — и колонка
  footstrap выродится в bootstrap. Это не регрессия роутера, а его контракт. Мерить
  надо на роутере без таких приложений (или помнить, что `spa_pages` в JSON перестал
  быть `38/38` именно поэтому — сверяйтесь с ним).
- `--ssh-host` — хост из `ssh -G` (по умолчанию `router`); роутер должен быть
  доступен по HTTP с машины запуска.
- `--runs N` — число замерных проходов (медиана). Меньше = быстрее, шумнее.
- `--out FILE` — сохранить результат в JSON.
- `--headful` — показать браузер (отладка).
- Скрипт **временно** переключает активную тему на роутере и в конце возвращает
  исходную (в т.ч. через `finally`). Если процесс убить `kill -9` — тема может
  остаться переключённой; вернуть вручную: `uci set luci.main.mediaurlbase=… ;
  uci commit luci`.

Время прогона ≈ `N_тем × (1 прогрев + runs) × N_страниц` навигаций — при `runs=3`,
трёх темах и 38 страницах это ~460 переходов, порядка 12–15 минут.

## Два бага, которые нашёл именно расширенный бенчмарк

Оба лежали в роутере с самого его появления и оба невидимы глазом — их находит только
измерение.

**1. Вьюха рендерилась дважды при первом заходе на страницу.** `require()` в LuCI
возвращает не класс, а **инстанс**: первый require конструирует объект, а `__init__`
вьюхи — это и есть её рендер (ровно это и делает `ui.instantiateView()`, больше в нём
ничего нет). Роутер делал `require()`, а затем строил поверх ещё один инстанс — страница
рисовалась дважды и вешала **два поллера**, то есть всё время работала с двойной частотой
RPC. Теперь новый инстанс строится только при **повторном** заходе, когда `require()`
отдаёт синглтон с уже отработавшим `__init__`. Тонкость: страница, пришедшая **полной
загрузкой**, тоже уже проинстанцирована LuCI, поэтому её класс засеивается в `_seen` при
инициализации — иначе первый же SPA-переход обратно на неё не отрисовал бы ничего.

**2. Поллер новой страницы ждал до 5 секунд.** LuCI держит **один** тик на секунду, а
`Poll.step()` запускает элемент очереди только когда `tick % interval == 0`. Полная
загрузка зовёт `Poll.start()`, тот ставит `tick = 0` и сразу делает `step()`. Роутер же
оставлял в живых тик **уходящей** страницы, поэтому поллер входящей вьюхи ждал ближайшего
кратного своему интервалу. Wireless рисует список станций из первого поллинга — и висел со
спиннером **4950 мс** против ~360 мс при полной загрузке. Теперь поллинг приводится в то
состояние, в котором его оставляет свежая загрузка (`stop()`, затем `start()` на **пустой**
очереди: `tick = 0`, таймер не взведён), а взводит таймер и делает первый шаг сам первый
`poll.add()` вьюхи. Это не обход апстрима — это буквально апстримная последовательность.
**Wireless: 4950 → 137 мс.**

Одного `Poll.stop()` мало и не было достаточно никогда: он **удаляет** `tick`, а
`Poll.add()` авто-стартует только при `tick != null` — страница не поллила бы вообще.

**И это один баг, а не два.** Перевзвод поллинга при всё ещё двойном рендере ронял
realtime-графики: `view/status/load.js` держит список графиков в **модульном** массиве
(модуль LuCI в SPA — кэшированный синглтон), шлёт по одному RPC на элемент и индексирует
ответы по **текущей** длине массива, — второй рендер дописывал массив прямо во время
запроса, длины расходились, и колбэк читал `datasets[gi]` как undefined. Починка двойного
рендера закрыла окно окончательно; сначала лечится корень, потом снимается симптом.

### Как читать

- **Сетевых запросов кардинально меньше**: 15–48 (bootstrap) и 27–72 (proton2025)
  против 0–8. SPA тянет только нужный RPC; полная перезагрузка каждый раз
  перезапрашивает каркас + скрипты (пусть и 304-ответами, но это раунд-трипы и
  повторный parse/exec). Ноль запросов у Administration/Password — вьюха уже в кэше,
  страница строится целиком из памяти.
- **Где выигрыш максимальный (8–17×)** — лёгкие вкладки и alias/firstchild-пункты. Там
  своей работы у страницы почти нет, и полная перезагрузка тратит время **целиком** на
  перезапуск рантайма. Ровно эти страницы старая выборка и не мерила.
- **Где выигрыш скромный (1.1–1.2×)** — Startup, Software, Overview: время доминирует
  сам **рендер контента** (парсинг init-скриптов, список пакетов, опрос ubus на
  дашборде), а не навигация. SPA убирает перезагрузку, но не ускоряет отрисовку view.
  Это потолок, а не недоработка.
- **Сторонняя тема может быть медленнее стоковой, которую перекрашивает**: proton2025
  проигрывает bootstrap (11680 против 10518 мс) — вдвое больше запросов и вчетверо
  больше CSS. Красивая тема не значит быстрая; это и есть аргумент за то, чтобы мерить.

Абсолютные числа зависят от железа (тут — filogic); важны **отношения** и падение числа
запросов.

### Как добавить тему в сравнение

Установить её на роутер (зарегистрировать в `luci.themes`) и дописать в `THEMES` в
`bench/nav-benchmark.py`. Кликом по ссылке ходит только footstrap — у остальных нет
клиентского роутера, и настоящая полная навигация это ровно то, что делает у них
клик. `BASELINE` задаёт, с чем считается колонка «ускорение».

# CSS: сборка из `styles/` и каскадные слои

До этого вся тема была одним файлом `cascade.css` на 4944 строки. Проблема была
не в размере файла, а в том, что он был устроен как хронологический changelog:
блоки назывались «audit fixes», «polish round 2», «review polish», один компонент
описывался в четырёх-шести разных местах (`.cbi-dropdown[open] > ul.dropdown` —
6 раз, `.btn`/`.cbi-button` — 5, `.tabs` — 4), а порядок между ними держался на
220 `!important`. Добавить правило можно было только в конец файла.

Теперь исходники разбиты по компонентам, а `cascade.css` **генерируется**.

Каталоги склеиваются в порядке `styles/` → `base/` → `theme/` → `pages/`, внутри
каждого числовой префикс — это и есть порядок конкатенации.

`base/` появился из одного файла на 2300 строк. Разрезание было чисто
механическим: порядок правил внутри слоя не изменился ни на строку, и `cssdiff`
по обоим layout'ам показал ноль расхождений — это и было условием, при котором
разрез считался выполненным.

## Слои

```css
@layer tokens, base, theme, page;
```

Поздний слой побеждает ранний **независимо от специфичности селектора**. Поэтому
правило из `theme/` не обязано перебивать правило из `base/` ни специфичностью,
ни `!important`.

Unlayered-уровень (правила вне всяких слоёв) сильнее любого слоя и **намеренно
оставлен пустым** — это аварийный люк. Туда же, кстати, попадает `node.css`,
который LuCI подключает для отдельных страниц после темы.

Почему палитры разрезаны на два файла: `03-palettes.css` содержит только
определения токенов и живёт в слое `tokens`, а правила вида
`:root[data-palette="rvht"] .fs-main { background-image: url(cats.svg) }` — это
обычные стили, и в слое `tokens` они бы проиграли слою `theme`. Они вынесены в
`theme/15-palette-rvht.css`.

### `!important` инвертирует порядок слоёв

Это неочевидно и это ловушка. Для important-объявлений порядок слоёв
**обратный**: `!important` в `base` бьёт `!important` в `theme`, а `!important` в
`theme` бьёт `!important` в `page`.

В `base/` осталось 20 `!important`, и часть из них выигрывает у темы, например
`.cbi-dropdown { display: inline-flex !important }`. Один флаг снят —
`border-color` в `input:focus, button:hover, …`: он рисовал чужое синее кольцо
поверх собственных цветов темы на каждом наведении и фокусе. Остальные держатся
не просто так: одиннадцать несут внутреннюю раскладку виджета `cbi-dropdown`
(снял пробно — дропдауны развалились на всех страницах), шесть — это утилиты
`.left/.right/.center/.top/.middle/.bottom`, чей смысл в принуждении, три бьют
inline-`style=` (цвета зон, SVG портов).

Практическое правило: если правилу темы нужен `!important`, чтобы победить
**другое правило footstrap**, то оно должно жить в более позднем слое, либо
оба правила надо слить. Флаг ставить не надо.

## Сборка

`build-css.sh` склеивает файлы и (без `--dev`) снимает комментарии, отступы и
пустые строки. **163 647 → 115 012 байт, −30 %.** Это не косметика: uhttpd отдаёт
`/www/luci-static/*.css` без gzip, поэтому каждый байт летит по проводу как есть.
Для сравнения, сток `luci-theme-bootstrap` — 44 КБ.

Трансформация не трогает ни селекторы, ни объявления. Собственный минификатор
LuCI (`csstidy`, `LUCI_MINIFY_CSS`) остаётся выключенным — он ломает `:has()` и
`color-mix()`.

Скрипт сам проверяет баланс скобок и отказывается писать битый файл.

### Как это работает на OpenWrt

`luci.mk` в `Build/Prepare` копирует в `PKG_BUILD_DIR` только пять каталогов —
`luasrc ucode htdocs root src` — и затем вызывает `Build/Prepare/$(LUCI_NAME)`.
Тема уже использовала этот хук, чтобы штамповать версию в JS. Туда же добавлена
сборка CSS:

```make
define Build/Prepare/luci-theme-footstrap
	$(SHELL) $(CURDIR)/build-css.sh $(PKG_BUILD_DIR)/htdocs/luci-static/footstrap/cascade.css
	$(if $(PKG_VERSION),$(SED) ...)
endef
```

`styles/` в список копирования не входит, поэтому читаем из `$(CURDIR)`, а пишем
сразу в build-дерево — исходники не пачкаются. Нужны только `cat` и `awk`, так
что билдбот OpenWrt соберёт это без единой host-зависимости.

Каталог **нельзя** называть `src/` — для `luci.mk` это C-исходники.

Сгенерированный `cascade.css` в git не хранится (`.gitignore`). `dev-sync.sh`
вызывает `build-css.sh --dev`, чтобы на роутере лежал читаемый файл.

### Как живут другие темы

Ни одна тема LuCI не гоняет препроцессор на билдботе — `luci.mk` умеет только
минифицировать (`csstidy`/`jsmin`), никакого `node`/`sass`/`postcss` там нет.
Поэтому argon (LESS), aurora (Tailwind v4 + Vite) и fluent (SCSS + Vite)
компилируют на машине разработчика и **коммитят готовый CSS**. Нам препроцессор
не нужен — хватает `cat`, и сборка честно едет в пакет.

Несколько `<link>` — тоже нормальная практика (сток bootstrap шлёт `cascade.css`
+ `mobile.css`), но выигрыша здесь нет: суммарные байты те же, а `?v=`
(`pkgs_update_time`) один на всю страницу, так что раздельно инвалидировать файлы
всё равно нельзя. Runtime-`@import` (как в material) хуже второго `<link>`: он
обнаруживается только после загрузки и парсинга первого файла, то есть
сериализует запросы и тормозит рендер.

## Проверка изменений: `cssdiff.py`

Скриншоты для проверки CSS не годятся. На живом роутере uptime, лизы DHCP и
уровень сигнала дают 0.5–1.3 % расхождения пикселей между двумя прогонами с
**одним и тем же** CSS, а настоящий регресс (кнопки сменили шрифт на
моноширинный) весит 0.19 %. Шум перекрывает сигнал.

`.claude/skills/footstrap-audit/cssdiff.py` грузит страницу один раз, снимает
`getComputedStyle` по 47 свойствам для каждого элемента, подменяет `<link>` на
вторую таблицу стилей и снимает снова. DOM тот же, данные те же — значит любая
разница вызвана только CSS.

```sh
scp -q old.css router:/www/luci-static/footstrap/cascade-a.css
scp -q new.css router:/www/luci-static/footstrap/cascade-b.css
LUCI_PW=… .claude/tooling/preview-venv/bin/python \
  .claude/skills/footstrap-audit/cssdiff.py \
  admin/network/firewall admin/system/system admin/status/overview admin/system/opkg
```

Он поймал четыре бага, которые скриншот-диф пропустил:

1. `.cbi-value-field *` красил моноширинным и кнопки внутри поля («Sync with
   browser»). Чинится через `:where(:not(.btn, .cbi-button))` — `:where()` держит
   специфичность на нуле, иначе правило начинает бить `<select>`.
2. `.cbi-dropdown { max-width: 100% }` стоял под баннером «phone overflow», но
   **вне** `@media (max-width: 600px)`. Он не проявлялся только потому, что более
   специфичное base-правило удерживало 400px.
3. Кнопки в строках таблиц описывались дважды — 28px с `!important` и 32px/12px
   ниже. Работал франкенштейн: высота от первого блока, паддинг от второго.
4. Колонка действий Edit/Delete теряла правое выравнивание.

### Ловушка в самом инструменте: веб-шрифты

Обе таблицы стилей объявляют одни и те же `@font-face`. Подмена `<link>`
перезапускает подбор шрифта, и если снять снимок до того, как шрифт готов,
измеряются метрики фолбэка — все ширины на странице уезжают на пиксель-другой и
топят настоящий диф. На firewall это давало 291 ложное расхождение. Лечится
ожиданием `document.fonts.ready` перед каждым снимком.

Отдельно: `admin/status/overview` сам себя опрашивает и перерисовывает секции,
поэтому даёт ~18 расхождений даже при одинаковом CSS с обеих сторон. Контрольный
прогон (A = B) показывает уровень шума конкретной страницы.

## Общий хром sidebar и top-nav

Различаться между вариантами темы должно только расположение меню. Brand, логотип,
wordmark, logout и индикаторы описывались **дважды** — без скоупа в
`20-shell-sidebar.css` и повторно под `.fs-topnav` в `26-topnav.css`, — и копии
разъехались: логотип 32px против 30px, зазор бренда 11px против 10px, зазор
индикаторов 10px против 8px.

Общий вид вынесен в `theme/10-chrome.css` со **значениями top-nav** как
эталонными. Layout-файлы теперь задают этим элементам только размещение (`flex`,
`order`, `justify-content`, схлопывание в icon-rail) — не внешний вид.

Критерий приёмки был жёсткий: на top-nav `cssdiff` обязан показать **ноль**
расхождений, а sidebar — ровно три заявленные дельты (логотип 32→30, зазоры
11→10 и 10→8). Так и вышло.

## Самостоятельность темы

Тема выросла из форка `luci-theme-bootstrap`, но реальной зависимости от
Bootstrap в ней почти не было. Разбор исходного `base` по категориям (388 правил,
33 КБ без комментариев):

| категория | правил | байт |
|---|---:|---:|
| формы и поля | 82 | 7 657 |
| виджеты LuCI (ifacebox, zonebadge, progressbar, tabs…) | 53 | 6 466 |
| reset + типографика | 69 | 3 952 |
| navbar / grid / прочий хром фреймворка | 34 | 3 323 |
| modal / alert / tooltip | 25 | 2 959 |
| виджет cbi-dropdown | 29 | 2 097 |
| кнопки | 23 | 2 559 |
| таблицы и секции | 29 | 1 582 |

То есть собственно фреймворк — это ~3.3 КБ навбара и сетки. Всё остальное —
**CSS виджетов самой LuCI**, который обязана поставлять любая тема: чекбоксы,
радио, dynlist, filebrowser, прогрессбары, зонные бейджи, тултипы, модалки.
Поэтому «избавиться» означало не выкинуть, а **присвоить**.

Сделано:

- navbar-хром (`header`, `.nav`, `.dropdown-menu`, `a.menu`, `.container`,
  `.pull-*`, `.tab-content`) — 13 правил, 3.2 КБ — удалён;
- остаток разрезан на `styles/base/*.css` по компонентам;
- все 28 захардкоженных цветов палитры 2012 года заменены на переменные моста
  (`audit.py` держит счётчик на нуле);
- слово «bootstrap» вычищено из имён файлов, комментариев и кода. Осталось ровно
  три места, и в каждом оно означает *другой, настоящий* пакет: fallback
  `/luci-static/bootstrap` в `uci-defaults`, базовая линия в `bench/` и блок
  атрибуции Apache-2.0 в `styles/00-header.css`.

Про атрибуцию: Apache-2.0 §4(c) требует сохранять уведомления об авторских правах
в производных работах. Пока в `styles/base/` живёт производный код, строки
«Copyright 2012 Nut & Bolt» и «Copyright 2011 Twitter, Inc» из баннера убирать
нельзя — это лицензионное обязательство, а не описание темы. Субъект баннера при
этом footstrap, а не фреймворк.

### Процедура присвоения

Она проверяемая, а не на глаз:

1. Удалить блок правил из `styles/base/`.
2. Прогнать `cssdiff` по обходу страниц в обоих layout'ах.
3. Ноль расхождений — тема уже покрывала этот блок, готово.
4. Есть расхождения — они и есть точный список того, чего теме не хватает.
   Дописать правила в нужный файл `styles/theme/` и повторить с шага 2.

На navbar-хроме `cssdiff` показал ровно три вещи, которых тема не покрывала и
которые молча наследовала: `line-height: 19px` у пунктов меню, `position:
relative` на `li` (к нему привязан каждый всплывающий подменю-попап) и тень
`<header>`. Они переехали в `theme/10-chrome.css` и `theme/50-topnav.css`. После
этого — ноль расхождений на обоих layout'ах.

### Свёртка дублей

Исходный `cascade.css` был устроен как хронологический changelog, и после разреза
по файлам это никуда не делось: правило дописывалось в конец файла вместо того,
чтобы отредактировать существующее. Счётчик — «сколько объявлений затенены другим
объявлением того же селектора в том же слое»: **182**, из них **108** дословно
повторяли уже заданное значение.

Хуже всего было в трёх файлах:

- `theme/40-tabs.css` описывал табы **дважды**: сначала как подчёркнутые, потом
  как таблетки в карточке. Второй блок выигрывал каждое свойство, первый был мёртв
  целиком. 162 строки → 104.
- `theme/55-buttons.css` задавал базовую кнопку в трёх местах, а сплошные варианты
  (`positive`/`action`/`negative`) — в двух, причём первый набор перекрывался
  вторым полностью.
- `theme/65-dropdown.css` описывал `.cbi-dropdown[open] > ul.dropdown` **пять раз**.

Результат: **182 → 0**, минифицированная сборка 115 795 → 110 456 байт.

Ловушка, на которой это едва не сломалось: в кнопках порядок правил нёс смысл.
`.cbi-button-save:hover { border-color: transparent }` и
`.cbi-page-actions .cbi-button-save { border: … }` имеют **одинаковую**
специфичность, так что рамка Save при наведении сохранялась только потому, что
второе правило написано ниже. То же и со сплит-кнопкой Save & Apply: её внешний
`div` — это и есть кнопка (`ui.js` эмитит
`.cbi-dropdown.btn.cbi-button.cbi-button-apply`), и правило «дропдаун в
`.cbi-page-actions` прозрачен» красило её прозрачным, пока следом не шло правило
`.cbi-button-apply`. При свёртке это выяснилось сразу — `cssdiff` показал
`background-color: accent -> transparent`. Теперь оба места выигрывают
**специфичностью**, а не позицией: `.cbi-page-actions .cbi-dropdown.cbi-button-apply`.

Порядок как несущая конструкция — это то же самое, чем раньше были 220
`!important`, только тише. `audit.py` теперь ловит такие дубли отдельной
проверкой, и группа-с-уточнением (`input…, textarea { min-height: 38px }` →
`textarea { min-height: 84px }`) от неё не срабатывает.

### Токенизация цветов

Литерал `#hex` не умеет следовать ни палитре, ни тёмной теме. Все 28 заменены на
переменные моста. `cssdiff` показал, что живым был ровно один — `header { color:
#BFBFBF }` в top-nav (теперь `var(--text-color-medium)`). Остальные 27 либо
перекрывались темой, либо сидели на селекторах, которых нет ни на одной странице
роутера, — и именно там прятались настоящие баги: `.close` рисовался `#000` на
тёмном фоне, `.cbi-section-error` — красным по `#fce6e6`, `.cbi-filebrowser` —
серым `#ccc`. Проверять такое нужно по `docs/gallery.html`, а не по роутеру.

### Галерея компонентов

`docs/gallery.html` рендерит каждый виджет с настоящими именами классов, которые
эмитят `ui.js`/`cbi.js`, — чтобы не искать по роутеру страницу, где случайно
встретится нужный контрол. Не входит в пакет; выкладывается рядом со стилем и
открывается без авторизации:

```sh
scp docs/gallery.html router:/www/luci-static/footstrap/
open http://<router>/luci-static/footstrap/gallery.html
```

Она сразу нашла две дыры, которых не было ни на одной странице роутера, но
которые может задеть любой сторонний `luci-app-*`: нативные
`input[type=button|submit|reset]` (`styles/base` давал им только
`width/height: auto`) и семейство `.label` (`.important`, `.warning`,
`.success`, плюс отсутствовавший `.danger`). Оба покрыты темой в токенах.

## Что поменялось визуально

Слой `base` — это не только «убрать `!important`». Правила темы, которые раньше
молча проигрывали по специфичности, теперь применяются:

- шапки `.ifacebox-head.active` («IPv4 Upstream», «radio0») стали темовыми
  вместо чужого синего;
- `.cbi-value-title` получил цвет `--dim`, как и было написано в теме;
- заголовки таблиц: паддинг 9→8px, начертание 700→600;
- строчные кнопки: 28px/10px (после слияния дублей).

Страница `admin/system/opkg` совпадает попиксельно.

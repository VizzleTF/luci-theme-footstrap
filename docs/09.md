# Хром темы: sidebar, рельс, бар

Спека того, что рисует `header.ut` и `menu-footstrap.js` (оба ссылаются сюда из
комментариев). Каскад и сборка CSS — docs/17, палитры — docs/08, SPA-роутер — docs/14.

## Раскладка — КЛИЕНТСКАЯ ось, а не запись темы

В `luci.themes` **одна** запись — `Footstrap` → `/luci-static/footstrap`. Sidebar и верхний
бар — одна и та же разметка и один рендерер; морфит их CSS по `:root[data-layout]`:

- атрибут **всегда несёт явное значение** (`sidebar` | `top`) и **штампуется сервером** в
  `partials/head.ut` (`<html data-layout=…>`), а инлайновый скрипт до первой отрисовки
  перекрывает его из `localStorage` `fs-layout`. Явное значение — не косметика: каждое
  layout-правило матчится **позитивно**, поэтому третья раскладка обязана будет опт-инить
  правила, а не наследовать сайдбаровы «просто потому, что она не top». Плюс хром корректен
  с выключенным JS. `:not([data-layout=…])` писать нельзя;
- переключение раскладки **ничего не перерендеривает**: DOM обслуживает обе, CSS меняет хром,
  а `MutationObserver` на `data-layout` в `menu-footstrap.js` складывает аккордеон в
  дропдауны и обратно;
- **миграция**: роутер, стоявший на старой top-nav теме, должен остаться с баром. Shell-скрипт
  не умеет писать `localStorage`, поэтому `uci-defaults` кладёт дефолт роутера в
  `luci.main.footstrap_layout=top`, `head.ut` его штампует, а собственный выбор пользователя
  перекрывает его навсегда. `postrm` ключ удаляет. Поэтому `applyLayout()` — единственная ось,
  которая пишет свой ДЕФОЛТ явно (`'sidebar'`, а не удаление ключа).

**Бар — это база, вертикальный сайдбар — исключение.** Бар написан один раз без гардов
(`styles/theme/20-shell.css`) и обслуживает верхнюю раскладку на любой ширине и сайдбарную —
на телефоне. Вертикальная колонка — единственный загарденный оверрайд, выигрывающий по
специфичности (`0,4,0` против `0,1,0`), а не порядком. `50-toplayout.css` — чистая дельта
десктопного бара. Подробности и причина инверсии — в CLAUDE.md.

## Целевая раскладка

```
┌──────────────────────────────────────────────────┐
│ .fs-sidebar 224px │ .fs-main (flex:1)            │
│ (rail: 68px)      │ ┌──────────────────────────┐ │
│  [лого] hostname ◂│ │ .fs-content:             │ │
│  #indicators      │ │  предупреждения          │ │
│  МЕНЮ             │ │  #tabmenu (вкладки)      │ │
│  ▸ Status ●       │ │  #view (контент)         │ │
│  ▸ System         │ │                          │ │
│  ▸ Network        │ ├──────────────────────────┤ │
│  …                │ │ footer.fs-footer         │ │
│  [spacer]         │ └──────────────────────────┘ │
│  Appearance [◐]   │                              │
│  ▸ Log out        │                              │
└──────────────────────────────────────────────────┘
```

Отдельного topbar внутри `main` нет: заголовок страницы — визуально скрытый `<h1>` внутри
`<div class="fs-title fs-sr">`, а `#indicators` (поллинг «Refreshing», несохранённые
изменения) живёт в сайдбаре. Кнопка `◂` (`#fs-rail-toggle`) сворачивает сайдбар в
68px-рельс с иконками и флайаутами.

**`.fs-sr` клипает, а не прячет.** Раньше здесь стоял `hidden` — это `display:none`, который
выкидывает элемент и из дерева доступности: `<h1>` фактически не существовал, а SPA-роутер
исправно обновлял в нём заголовок, которого не видел ни один скринридер. Клип (как у
`.fs-skip`) оставляет элемент в дереве. Рядом лежит `#fs-nav-status` (`role="status"`,
`aria-live="polite"`) — роутер пишет туда заголовок после каждой SPA-навигации, иначе смена
страницы без перезагрузки для скринридера не происходит вовсе.

## Маппинг LuCI-контейнеров

Меню рендерит клиентский JS в фиксированные контейнеры (docs/01, docs/04):

| LuCI-контейнер | В bootstrap | В footstrap |
|---|---|---|
| `#topmenu` | горизонт. меню верхних разделов | `ul.nav` в `.fs-sidebar`: вертикальный список / бар / флайауты рельса — **одна разметка** |
| `#modemenu` | breadcrumb-переключатель admin/status | `.fs-modemenu` в сайдбаре; пустой/единственный режим → `display:none` |
| `#tabmenu` | вкладки под шапкой | остаётся в main, над `#view` (контейнер эмитит `partials/notices.ut`) |
| `#indicators` | правый угол шапки | `.fs-indicators` в сайдбаре под брендом; пустой → скрыт |
| `#view` | контент | контент (main column) |

Набор верхних разделов приходит из дерева меню и зависит от установленных пакетов —
**не хардкодить**.

## `data-narrow` — «сайдбар стал баром», и это ИЗМЕРЕНИЕ

Сайдбар уступает место бару, когда оставшейся контентной колонке уже нечего показать.
Вьюпортный брейкпойнт этого сказать не может: срез — не константа (224px развёрнутый,
68px рельс), так что один брейкпойнт давал обоим состояниям один ответ, и рельс складывался
на той же ширине, что и полный сайдбар.

`fitShell()` (`fs-chrome.js`) вычитает срез из `innerWidth` и сравнивает остаток
с `--fs-content-min`; ширины он **читает из CSS-токенов** (`--fs-sidebar-w`, `--fs-rail-w`,
`--fs-content-min`, `--fs-content-pad`) через `getComputedStyle`, а не хранит копией — иначе
сужение рельса в стилях оставило бы измерение вычитать старое число, и ни один гейт этого
не заметил бы. Результат — атрибут `data-narrow` на `:root`: на него смотрит и CSS, и
`flyoutMode()` в JS.

Измерение, наблюдатель и коалесинг живут в `fs-fit.js` — одном движке «влезает ли ещё?»
на всю тему (его же использует `fitTables` в `fs-select.js`). Новую логику «влезает или нет»
добавлять туда, второго наблюдателя не заводить.

Единственный литерал — `@media (min-width: 521px)` вокруг вертикального оверрайда в
`20-shell.css`: пол, ниже которого никакой срез не оставит читаемых 500px, и заодно
страховка при выключенном JS (нет JS — нет `data-narrow`, и телефон отрисовал бы десктопный
сайдбар).

## Файлы

### `ucode/template/themes/footstrap/header.ut`

- Общая часть — в `partials/` (`head`, `brand`, `appearance`, `logout`, `notices`, `footer`).
  Второго каталога шаблонов нет.
- `<body>` → обёртка `.fs-shell` (flex). Перед ней — skip-link `.fs-skip`
  («Skip to content» → `#maincontent`), первый таб-стоп страницы.
- **Sidebar** — `<nav class="fs-sidebar" aria-label="Menu">` (именно `<nav>`: `<aside>` даёт
  роль `complementary`, и по лендмарку до меню было не допрыгнуть):
  - `.fs-brandrow`: бренд (градиентный квадрат + wifi-SVG на `currentColor` + wordmark
    hostname/OpenWrt) + кнопка `#fs-rail-toggle`;
  - `<div id="indicators">`;
  - `.fs-navlabel` («Menu») + `<ul class="nav" id="topmenu">` (наполняет menu-JS) +
    `<ul id="modemenu">`;
  - spacer `.fs-spacer` (`flex:1`);
  - кнопка Appearance (`#fs-appearance`) и Log out.
- **Main**: `<main class="fs-main" id="maincontent" tabindex="-1">` — `.fs-title.fs-sr` с `<h1>`,
  `#fs-nav-status` и `.fs-content` (предупреждения, `#tabmenu`, `#view`, `footer`).
- `data-page` несёт **dispatch-путь** (`ctx.path`), а не `request_path`: на firstchild-роуте
  `/admin/status` рендерится overview, а `request_path` — только `['admin','status']`, и все
  правила `body[data-page='admin-status-overview']` (и overview-include, который на этом
  гейтится) молча не применялись.
- Обязательное к сохранению: `http.prepare_content`, `cbi.js`, переводы, `node.css`, `css`,
  `blank_page`, noscript, предупреждения (no-password, initramfs) — docs/03, docs/06.

### `sysauth.ut` — тема несёт свой, и это обязательно

Он существует ради одной вещи: передать `blank_page: true`. Без темо-локального `sysauth`
диспетчер берёт generic-шаблон, который инклюдит header **без** `blank_page`, — и весь хром
(сайдбар, меню, футер) рисуется вокруг формы логина, а все его контролы мертвы.

Это **не** копия bootstrap-овского `sysauth.ut`: тот прячет форму в `<section hidden>` и
раскрывает её view-модулем — здесь это дало **пустую страницу без возможности войти** (view
поднимается до того, как есть сессия, его RPC отвечают «Access denied», промис реджектится,
`render()` не вызывается). Форму рисует **сервер**: работает с выключенным JS, не ломается
реджектнутым промисом, не требует установленного `luci-theme-bootstrap`. Оформление —
`styles/pages/10-login.css`.

### `htdocs/luci-static/resources/menu-footstrap.js` — ЕДИНСТВЕННЫЙ рендерер

- `renderMainMenu` → список в `#topmenu`: пункт =
  `<li><a><icon><span class="fs-label">title</span><chevron></a></li>`, активный — по
  `L.env.dispatchpath`. Иконки — map по имени раздела (`status`, `system`, `network`, `vpn`,
  `docker`, …) с regex-фолбэком и generic-SVG; SVG инлайн в JS. Всё общее (табы, режимы,
  поповер Appearance, рельс, SPA-роутер) — в модулях `fs-*.js`, которые разводит
  `menu-footstrap-common.js`; layout-специфичен
  только `renderMainMenu`, он передаётся в `common.init()` (композиция, а не наследование:
  LuCI делает из каждого baseclass синглтон).
- Верхний узел `admin/logout` из дерева выкидывается — свой Log out хром рисует сам
  (`partials/logout.ut`), иначе он двоится.
- Раздел с детьми — **disclosure-паттерн W3C APG**, а не ссылка: `role="button"`,
  `aria-expanded`, `aria-controls`, Enter/Space, Escape закрывает флайаут и возвращает фокус.
  Осознанно **не** `role="menu"`: APG прямо запрещает навигации сайта брать семантику menubar.
  `aria-current="page"` — только на листе; заголовок секции это кнопка, а не ссылка на
  текущую страницу.
- Два смысла `.open`: в развёрнутом сайдбаре — аккордеон (несколько секций сразу, набор
  помнится в `localStorage` `fs-menu-open`), в рельсе / баре / на узком экране — эксклюзивный
  флайаут. Решает `flyoutMode()`, и он читает **ровно то же, что стилевой лист**:
  `data-rail` / `data-layout=top` / `data-narrow`. **Никогда не вьюпортный брейкпойнт** —
  раньше там был `matchMedia('(max-width: 767px)')`, и в окне 768–779px хром был баром, а меню
  считало себя аккордеоном: Escape и клик снаружи не закрывали панель, `clampDropdown` её не
  ставил. На выходе из флайаут-режима аккордеон восстанавливается (`restoreAccordion()`) —
  простого `closeFlyouts()` мало: он снимал `.open` со всего, а разметка на переключении
  рельса не пересобирается, и «Keep open» переставал что-либо значить.
- `clampDropdown` — подвинуть дропдаун обратно в вьюпорт у правого края. Переехал сюда из
  удалённого top-nav рендерера; это была единственная уникальная логика, которую тот нёс.
  Только для десктопного бара (`topBarMode()`: `data-layout=top` + не мобильный `@media`,
  под который загарденa дельта `50-toplayout.css`) — на телефоне панель якорится к левому
  краю бара, в рельсе улетает вбок. Держит по одному запланированному `rAF` **на `<li>`**,
  чтобы **отменять** незавершённый замер, когда указатель уехал дальше; общий коалесер
  `fit.frame()` этого выразить не может — это задокументированное исключение.
- Hover-открытие — чистый CSS.

### `menu-footstrap-common.js` — bootstrap общего хрома

> Сам файл теперь только грузит дерево меню и разводит модули: рендер табов и mode-меню, рельс и
> замеры — в `fs-chrome.js`, SPA-роутер — в `fs-router.js`, оси — в `fs-prefs.js`, поповер — в
> `fs-appearance.js`, примитивы disclosure — в `fs-widgets.js`. См. CLAUDE.md, «The JS modules».


Поповер **Appearance** (`#fs-appearance`) — все оси клиентские, из `localStorage`, без сервера
и перезагрузки: **Layout** (sidebar/top), **Theme** (auto/light/dark), **Palette**
(footstrap/hicontrast), **Wallpaper** (off/cats), **Tint** (оттенок 1–360° для «узнать этот
роутер», красит холст), **Accent** (оттенок 1–360° для акцентных контролов), **Rounding**
(0–20px), **Submenus** (Keep open / Auto-collapse; в баре скрыт CSS, но строится всегда),
**Updates** (проверка новой версии) + строка версии и кнопка Update.

Каждая ось реализована **дважды**: `partials/head.ut` штампует `:root` до первой отрисовки
(инлайн, до модульного загрузчика — `require` там невозможен), а common.js применяет её живьём.
Убрать ни одну копию нельзя, байт-в-байт они не совпадут, поэтому контракт (ключи, атрибуты,
свойства, диапазоны, дефолт скругления и **порядок «сначала custom property, потом атрибут»**)
держит гейт `tools/axes.mjs` (`npm run axes`). Порядок несущий: наоборот — один кадр после
перезагрузки красится предыдущим оттенком.

Каждая подпись в поповере несёт `msgctxt` `footstrap`: msgid — **глобальное** имя, LuCI грузит
все `*.<lang>.lmo` в один каталог, и порядок `readdir` решает, чей перевод победит. Без
контекста переключатель раскладки рисовал на русском роутере «Максимум» (issue #6): чей-то
каталог переводит msgid `Top` как «максимум». Хром (Menu, Logout, Skip to content) и фразы
логина/предупреждений оставлены **без** контекста намеренно — унаследовать перевод `luci-base`
в ~40 языках, для которых у темы каталога нет, это фича.

Контракт `data-darkmode` (docs/04) держим; наружу дополнительно пишутся `data-theme` /
`data-bs-theme` — их нюхают сторонние приложения (justclash, ssclash), внутри темы их не читает
никто. `auto` слушает `matchMedia('(prefers-color-scheme: dark)')`.

## Шрифты и CSS

Самохостинг (роутер офлайн): Manrope 600/700 и JetBrains Mono **400**, каждое начертание
разбито по `unicode-range` на latin / latin-ext / cyrillic; `@font-face` — в
`styles/01-fonts.css`. Латинские сабсеты Manrope предзагружаются в `partials/head.ut` — ими
рисуется сам хром; JetBrains Mono не предзагружается (он нужен в ячейках таблиц много позже
первой отрисовки).

**Жирного моноширинного нет и быть не должно.** `<strong>` — это ЛЕЙБЛ (LuCI пишет статусы как
`<strong>MAC:</strong> ac:1f:6b:…`), поэтому на моноширинных поверхностях он берёт UI-начертание
(`theme/45-misc.css`). До этого правила 227 элементов на семи страницах рисовались жирным моно,
и браузер тянул `jetbrains-mono-600` (20 КБ) ради слова «MAC:» — 30% всего шрифтового бюджета.

`cascade.css` **генерируется** `build-css.sh` из дерева `styles/` и в git не лежит; слои —
`tokens, base, theme, page`; два яруса токенов (приватные `--fs-*` читает вся тема, экспортные
`--*-color-*` — только сторонние `luci-app-*`). Всё это — docs/17 и CLAUDE.md, здесь не
дублируется.

# SPA-роутер (мгновенная навигация без перезагрузки)

Как `luci-theme-footstrap` убирает полную перезагрузку страницы при клике по
меню. Реализация — вариант C из ресёрча: клиентский роутер только для
`view`-узлов, с безопасным фолбэком в обычную навигацию для всего остального.

Живёт в `htdocs/luci-static/resources/fs-router.js`; разрешение пути в узел меню — в
`fs-menutree.js` (его же читает хром), защита от чужого инжектнутого CSS — в `fs-sheets.js`. Правок
сервера / luci-base / шаблонов **нет** — это чистый аддитивный theme-JS.

Этот документ — про то, **как роутер устроен**. Про то, **как правильно** (History/Navigation
API, отмена летящей работы, гонки, teardown, скролл, a11y — со ссылками на спеку) и чему из
этого мы не соответствуем, с замерами на живом роутере — `docs/22`.

## Почему это вообще возможно

LuCI 25.12 — классический MPA: каждый клик = полный HTTP GET → ucode-диспетчер
заново отдаёт весь каркас страницы. Но **контент и так рисуется на клиенте**:
диспетчер для `view`-узла рендерит шаблон `view.ut`, который эмитит
`<div id="view">` + инлайн-скрипт `L.require('ui').then(ui => ui.instantiateView(path))`.
Класс `LuCI.view` (см. `luci.js`) в `__init__` берёт `#view`, ждёт `luci-loaded`,
затем `load()` → `render()` → вставляет ноды в `#view`.

То есть сервером управляется только *навигация*, а не *рендер*. Роутер повторяет
ровно то, что делает `view.ut`, минус перезагрузка: перехватывает клик,
переинстанцирует нужный view в существующий `#view`, обновляет URL через
`history.pushState`.

Дерево меню (`ui.menu.load()`, кэш в `sessionStorage`) даёт по каждому узлу
`action.type` и `action.path`. Считать надо не листья, а **кликабельные узлы** —
те, что `ui.menu.getChildren()` вообще превращает в `<a href>` (`satisfied` +
есть `title`): узел с `action` может иметь детей (`admin/status/nftables`), а
`rpc/*`, `admin/uci/*`, `admin/menu` титула не имеют и ссылками не становятся
никогда. На тестовом роутере таких узлов **65**, и SPA обслуживает **62**:

| тип | сколько | SPA |
|---|---|---|
| `view` | 54 | да |
| `alias`, `firstchild` | 10 | да — резолвятся в лист (ниже) |
| `template` (overview) | 1 | да — спецслучай |
| `call` (`admin/vpn/openvpn`, Lua-CBI) | 1 | нет — рендерится сервером |
| `function` (`admin/logout`) | 1 | нет — и не нужно |
| `firstchild` → в `call` (`admin/vpn`) | 1 | нет — резолвится в не-SPA-able узел |

Итого не-SPA только то, у чего **нет клиентского view-класса в принципе**:
серверный обработчик Lua-эпохи и логаут.

## alias / firstchild — резолв на клиенте

Это была слепая зона роутера: 7 из 27 ссылок, которые рисует меню (Firewall,
System Log, Realtime Graphs, Administration, Terminal, Attended Sysupgrade,
Justclash), — не страницы, а редиректы. `viewClassFor` видел не-`view` и уходил в
полную перезагрузку, то есть **самые нажимаемые пункты меню перезагружали страницу**.

Сервер их не редиректит: GET `/admin/status/logs` отвечает **200 по тому же URL**,
а лист разрешает внутри — `pathinfo` остаётся запрошенным путём, а
`requestpath`/`dispatchpath`/`nodespec`/`ctx.path` уже несут `…/logs/syslog`
(проверено на живом роутере). Клиент делает то же самое:

- `resolveSegs()` идёт по `alias` (переход на `action.path` от корня) и по
  `firstchild` (выбор ребёнка), пока не упрётся в настоящий узел; счётчик прыжков
  ловит цикл в чужом `menu.d`.
- `firstChildOf()`/`nodeWeight()` — **перенос** `resolve_firstchild()`/`node_weight()`
  из `dispatcher.uc`, не пересказ: вес `min(order ?? 9999, 9999)` плюс 10000 для
  узла с `auth.login`, фильтры `satisfied`/`title`/`firstchild_ineligible`,
  рекурсия во вложенный `firstchild` (он годится, только если сам во что-то
  разрешается), ничья → первый по порядку ключей. ACL-проверку пропускаем: дерево,
  которое клиент получает из `/admin/menu`, уже отфильтровано по ACL сессии.
- `navigate()` держит **обе** дорожки: `segs` (что кликнули) идёт в `pushState` и
  `pathinfo`, `rsegs` (разрешённый лист) — в `requestpath`/`dispatchpath`/`nodespec`/
  `data-page`/заголовок. Ровно так же, как при полной загрузке.

Точность здесь не эстетика: выбери клиент другого ребёнка — по клику открывалась
бы одна страница, а по F5 того же URL другая. `rewrite` намеренно не резолвится
(в дереве его нет, а ошибка в семантике `splice` открыла бы **не ту** страницу —
это хуже, чем перезагрузка, в которую она проваливается).

## Поток навигации

`wireRouter()` вешает один делегированный обработчик на `document`:

1. Клик по `<a href>`, без модификаторов (ctrl/meta/shift/alt), кнопка 0, не
   `target=_blank`, не `download`, тот же origin, href не `#…`. Ссылки с
   `?query` или `#hash` тоже уходят в полную загрузку: `navigate()` носит только
   pathname, а `pushState` голого пути срезал бы и то и другое (а view читает
   `location.search`). По той же причине `popstate` на запись с query — просто
   `location.reload()`.
2. `navigate(pathname, push=true)`:
   - `segsFromPath` срезает `L.env.scriptname` → массив сегментов пути;
   - `documentPoisoned()` — не отравил ли документ инвазивный CSS чужой вьюхи
     (см. «Сторонний CSS вьюхи»); отравил → `return false`;
   - `nodeForSegs` идёт по дереву меню до узла, `viewClassFor` даёт имя класса
     view (или `null`, если узел не SPA-able);
   - **если класса нет (не `view`/overview-`template`, или `satisfied === false`,
     или нет `.fs-content`, куда вставить `#view`) → `return false`** → обработчик
     НЕ делает `preventDefault` → браузер грузит страницу как обычно (фолбэк);
   - иначе: teardown → обновление `L.env` → `body[data-page]` → `pushState` (а если
     кликнули по **уже открытой** странице — `replaceState`: вторая запись сделала бы
     одно нажатие Назад мёртвым) → `renderChrome()` → `scrollTo(0, 0)` → фокус на
     `#maincontent` + объявление нового заголовка в polite-live-регион
     (`#fs-nav-status`) — то, что при полной загрузке делает сам браузер, а SPA обязан
     сделать руками: `renderChrome()` только что снёс `<a>`, который пользователь нажал
     Enter'ом, и фокус упал бы в `<body>` молча → переинстанцирование view;
   - `return true` → `preventDefault`.
   Каждая закоммиченная навигация инкрементит `_navGen`: два быстрых клика
   гоняют свои async-`require()`, и без токена поколения первый view мог бы
   отрисоваться в `#view` уже после второго.
3. `popstate` (back/forward): `navigate(location.pathname, push=false)`; если
   узел не SPA-able → `location.reload()`. Две защиты перед этим: запись с
   `?query` — сразу `location.reload()` (мы пушим только голые пути, и replay
   голым путём потерял бы query, который читает вьюха); и **смена фрагмента — не
   навигация**: Chrome шлёт `popstate` на same-document переход по `#`, поэтому
   клик по `<a href="#">` внутри вьюхи (частая идиома in-page-контролов) приходил
   сюда как «нажали Назад», и роутер переинстанцировал вьюху, стирая состояние,
   которое клик только что поставил (issue #3: таб-полоса `luci-app-filemanager` —
   четыре `<a href="#">`, чей обработчик не зовёт `preventDefault`). Если
   `location.pathname === _curPath` — страница владеет фрагментом сама, выходим.

Так как `pushState` кладёт **реальный** URL диспетчера, F5 и deep-link работают
серверно без изменений.

`document.body[data-page]` роутер перештампывает сам — из **резолвнутого**
пути листа (`rsegs.join('-')`), ровно как сервер штампует `ctx.path` при полной
загрузке. Иначе входящая страница осталась бы с `data-page` предыдущей, и
page-scoped CSS (`styles/pages/*`) молча не применился бы; заодно
firstchild-URL вроде `/admin/status` даёт тот же `admin-status-overview`, что и
полная загрузка.

## Переинстанцирование view — главная тонкость

**Почему** `require()` на первом заходе уже отрисовал страницу и `new` после него
делать нельзя (двойной рендер, два поллера), и зачем нужен `_seen` — разобрано в
[docs/15 → «Баг 1»](15-benchmark-navigatsiya.md); там же засев `_seen` страницей,
пришедшей полной загрузкой. Не дублируем: это один и тот же баг.

Механика, которой нет там. При **повторном** заходе require отдаёт синглтон, чей
`__init__` уже отработал, — повторный вызов ничего не перерисует. Класс достаётся из
инстанса: система классов LuCI ставит `ClassConstructor.prototype.constructor =
ClassConstructor`, поэтому `instance.constructor` — это класс.
`new instance.constructor()` запускает свежий `__init__` → свежие `load()`+`render()`
в `#view`. Это идентично полной загрузке, которая тоже всегда стартует с нового
инстанса.

## Ловушка: два разных `L` (была причиной бага `L.itemlist is not a function`)

`L` внутри модуля (параметр фабрики) и `window.L` (рантайм-инстанс, который
диспетчер создаёт как `new LuCI()`) — **разные объекты**. Модуль `ui` навешивает
хелперы (`itemlist`, `showModal`, `hideTooltip`, …) именно на `window.L`, а не на
прототипный `L`, который получают фабрики. Требуемый модуль захватывает тот `L`,
на котором вызвали `require()`.

Поэтому view **обязан** требоваться через `window.L.require(...)` — иначе он
захватит прототипный `L` без хелперов и упадёт в середине рендера на первом же
`L.itemlist(...)`. В коде это `const RT = window.L; RT.require(className)…`.

`L.env` и `L.Poll` при этом общие (замыкание/синглтон), так что менять их можно
через любой `L`; критичен только таргет `require`.

## Teardown

Перед рендером нового view:

- **Поллинг приводится в то состояние, в котором его оставляет свежая загрузка** —
  все три шага обязательны:

  ```js
  L.Poll.queue.length = 0;   // поллеры уходящего view
  L.Poll.stop();             // снять его tick
  L.Poll.start();            // на ПУСТОЙ очереди: tick = 0, таймер не взведён
  ```

  *Flush* — чтобы поллеры ушедшей страницы не били по detached-DOM и не жгли RPC.
  Единственный не-view-поллер, который добавляет LuCI, — транзиентная проверка
  достижимости при apply/reboot, так что чистить очередь безопасно.

  *Одного flush мало.* LuCI держит **один** тик в секунду и запускает элемент
  очереди только при `tick % interval == 0`. Оставленный в живых тик **уходящей**
  страницы заставлял поллер входящей вьюхи ждать ближайшего кратного своему
  интервалу — до 5 с (`pollinterval`). Wireless рисует список станций из первого
  поллинга и висел со спиннером **4950 мс** против ~360 мс при полной загрузке
  (нашёл бенчмарк, [docs/15](15-benchmark-navigatsiya.md), баг 2).

  *Одного `stop()` тоже мало* — и это причина, по которой шагов три: `stop()`
  **удаляет** `tick`, а `Poll.add()` авто-стартует только при `tick != null`, то
  есть страница не поллила бы вообще. `stop()` + `start()` на **пустой** очереди
  даёт ровно то, что есть в свежем документе: `tick = 0`, таймер не взведён
  (`start()` на пустой очереди его не взводит). Взводит его и делает первый шаг
  немедленно — первый же `poll.add()` входящей вьюхи. Это не обход апстрима, а
  буквально апстримная последовательность: при полной загрузке `initDOM()` зовёт
  `Poll.start()` на пустой очереди ещё до рендера вьюхи.
- Индикатор «Refreshing»/«Paused» переживал собственный поллинг: LuCI показывает
  его на `poll-start`, переключает на `poll-stop` и больше не прячет никогда, а
  наш `stop()` на каждой навигации `poll-stop` **диспатчит** — переход с
  поллящейся страницы на не-поллящуюся оставлял «Paused» о несуществующем
  поллинге. Свой листенер `poll-stop` (регистрируется на eval модуля, т.е. после
  луциевского, и потому отрабатывает вторым) прячет пилюлю, когда очередь пуста.
- `clearViewIntervals()` — прибить голые `window.setInterval` уходящего view
  (например log-tailer у luci-app-podkop): полная загрузка убила бы их сама, SPA
  обязан явно. `setInterval`/`clearInterval` перехвачены на этапе eval модуля,
  id'шники трекаются в `Set`; собственный 1-секундный tick `L.Poll` сохраняется.
- отмена собственной self-update-цепочки: `_updTimer` (raw `setTimeout`)
  очищается, и `_updGen++` **разотождествляет уже летящий** `fs.exec` — таймера
  в этот момент нет (rpctimeout 20 с — окно широкое), и без токена поколения
  ответ перепланировал бы следующий тик и мог выкинуть свою модалку поверх
  посторонней страницы.
- `ui.hideModal()` — закрыть возможную модалку.

## renderChrome()

После смены `L.env` (`requestpath`/`dispatchpath`/`pathinfo`/`nodespec`)
перестраивает mode-меню + основное меню + вкладки секций из нового `L.env`.
Контейнеры (`#modemenu`/`#topmenu`/`#tabmenu`) чистятся перед ре-рендером, чтобы
не дублировать. Обновляются также `document.title` и `.fs-title-main`.

## Границы и деградация

- Раскладка роутеру безразлична: рендерер **один** (`menu-footstrap.js`,
  `common.init(renderMainMenu)` зовётся ровно один раз), а sidebar/top —
  клиентский `:root[data-layout]`, который CSS морфит и которого роутер вообще не
  касается.
- Сторонние приложения, регистрирующие `view`-узлы, ускоряются автоматически.
- Приложение, инжектящее **инвазивный CSS**, намеренно роняет навигацию в полную
  загрузку — см. следующий раздел.
- Legacy `cbi` и обработчики `call`/`function` → полная навигация.
- **Status→Overview** (`template`-узел `admin_status/index`) — SPA-исключение:
  его серверный шаблон лишь определяет 3 глобал-хелпера
  (`progressbar`/`renderBox`/`renderBadge`) и зовёт `ui.instantiateView('status/index')`.
  Роутер это воспроизводит: `ensureOverviewHelpers()` идемпотентно определяет
  хелперы (инлайн-скрипт шаблона при SPA не выполняется), затем инстанцирует
  `view.status.index`. Прочие `template`-узлы → полная навигация.
- Любая ошибка require/instanceof → `console.error(...)` и `window.location =
  pathname` (полный переход). Ошибку именно **логируем**: молчаливый фолбэк
  делал любую регрессию роутера похожей на «страница просто медленно грузится».
- Другие темы не затрагиваются.

## Сторонний CSS вьюхи: гейт, а не подметание

`<style>`/`<link>`, который вьюха вписала в `<head>`, умирает вместе с документом при
полной загрузке — но **переживает SPA-переход** и красит все страницы после.
`luci-app-filemanager` инжектит
`.cbi-button-apply, .cbi-button-reset, .cbi-button-save:not(.custom-save-button)
{ display: none !important }` — unlayered + `!important`, то есть побеждает любой
каскадный слой: один заход, и Save/Reset исчезли со **всех** config-страниц.

**Просто удалять на навигации нельзя** — пробовали, сломали SSClash. Поллер
восстановим ре-рендером вьюхи; стиль вернётся, только если инжектор запустится
**снова**, а библиотека, импортирующая CSS на eval модуля, не запустится никогда
(модуль кэширован на жизнь документа). `ace_editor.css` (14 KB абсолютно
спозиционированных слоёв) импортируется один раз — после подметания возврат в
редактор давал чёрный прямоугольник высотой 2 007 346 px. Удаление молча
одностороннее.

Отсюда конструкция: **`documentPoisoned()` перед каждой навигацией**. Есть в документе
инвазивный лист → `navigate()` возвращает `false` → обычная полная загрузка. Меняем
скорость на корректность, никогда наоборот; свежий документ никакого view-CSS не несёт,
так что SPA работает дальше сразу же.

`VIEW_SHEETS` = `style:not([data-fs-shell]), link[rel~="stylesheet"]:not([data-fs-shell])`.
`<link>` — не гипотеза: `luci-app-banip`/`luci-app-adblock` дописывают
`<link …/custom.css>` в `<head>` на eval модуля, и он красит `.cbi-input-text` /
`.cbi-input-select` — штатные виджеты, на каждой странице, unlayered. Исключены:
`[data-fs-shell]` (единственный `<style>`, который эмитит сервер, — **помечен**, а не
угадан) и всё внутри `#view` (умирает вместе с подменой контента). Ядро LuCI в рантайме
`<style>` не инжектит вообще (проверено: `luci.js`, `ui.js`, `cbi.js`).

**Вселенная имён темы читается из самого `cascade.css`** (`themeNames()`; same-origin,
поэтому `cssRules` читаемы) — все классы/id, которые тема стилизует, и все custom
properties, которые она объявляет или читает. Не рукописный список: тот отстанет от
темы в первый же день. Нечитаемый лист (ещё грузится, 404, cross-origin) или
нечитаемый `cascade.css` = **инвазивен по умолчанию**: неизвестный CSS уходит на
медленный путь, но не на сломанный. Цена всего гейта — ~0.3 мс на навигацию.

Три проверки в `invasiveSheet()`:

1. **Голый селектор типа** (`pre`, `*`, `:root`, `svg text` — ни класса, ни id, ни
   атрибута) матчит штатную разметку на любой странице → инвазивен. **Если** его
   объявления не инертны: `inertDeclarations()` пропускает правило, объявляющее
   только custom properties, которых тема **не читает** — покрасить нас оно не может.
   `luci-app-temp-status` открывается с `:root { --app-temp-status-temp: #147aff; … }`
   и по одному селектору был бы «документ отравлен». Инвазивны: любое **стандартное**
   свойство на голом селекторе (штатный filemanager пишет `:root { color-scheme: light
   dark }` — перенацеливает все UA-виджеты на настройку ОС) и любая custom property,
   которую тема **читает** (в этом и смысл приватного `--fs-*`-тира: чужой `:root` с
   `--accent`/`--radius` нас не перекрашивает — и не должен начать).
2. **Имя темы без якоря.** Правило может называть штатный виджет и всё равно быть
   безвредным, если может сматчиться только внутри разметки своего приложения:
   `#cbi-podkop-section > .cbi-section-remove` требует секции podkop. Прибивает его к
   месту имя, которого тема **не знает**, — собственное имя приложения. Селектор,
   целиком собранный из знакомых теме имён, ничем не прибит и матчит те же виджеты на
   любой чужой странице → инвазивен.
3. **Аргументы функциональных псевдоклассов срезаются** перед поиском якоря — и в этом
   вся разница между podkop и filemanager: `.cbi-button-save:not(.custom-save-button)`
   тоже называет класс приложения, но **внутри отрицания** — он не требует его
   разметки, он её исключает.

**Единственное, что удалять безопасно, — байт-в-байт вторая копия.** Не-удаление стоит
дорого там, где приложение инжектит на **каждый** рендер: podkop зовёт
`injectGlobalStyles()` из `render()` (4 KB, без guard'а), `luci-app-mosdns`
переприлепляет три `<link>` CodeMirror — каждый SPA-визит добавляет копию, которую
браузер парсит вечно. Точный дубль сносить можно ровно по той причине, по которой
подметание ломалось: правила никуда не деваются — уцелевшая копия байт-в-байт та же, и
проверка библиотеки «а я это уже импортировал?» (на чём умер ACE) по-прежнему находит
свой sheet. Оставляем **первую** копию: на неё указывают хендлы, которые приложение
сохранило. `watchViewSheets()` вешает наблюдателя на `<head>`, а не подметает на
навигации: podkop инжектит из `render()`, которое резолвится **после** колбэка
`require()` роутера, поэтому подметание на навигации оставляло документ вечно нести одну
устаревшую копию. Зациклиться наблюдатель не может: удаление даёт мутацию без
**добавленных** узлов, а обработчик выходит, если стиль не добавлялся.

## Быстрый двойной клик по НЕкэшированным вьюхам

`_navGen` отменяет устаревшую навигацию — но только на **кэшированном** пути: там
отмена и есть «не звать `new view.constructor()`». На **первом** заходе `require()`
**и есть** рендер: он конструирует вьюху, чей `__init__` делает `load()` → `render()` →
`dom.content(#view)` и регистрирует поллеры, — внутри промиса, который нам не
принадлежит. Отменять нечего.

Поэтому быстрый двойной клик — настоящий баг: клик Firewall (не в кэше), через 100 мс
клик Wireless. `navigate(Wireless)` чистит очередь `L.Poll` **до** того, как Firewall
добавит туда свой поллер; Firewall дорисовывается в `#view`, который уже принадлежит
Wireless, и вешает поллер, которого флаш уже не поймает. Остаётся: URL/заголовок/меню/
`data-page` от Wireless, контент от Firewall и поллер Firewall — навсегда, на всех
следующих страницах.

Чинится **повторной навигацией**: `repairStaleRender(className)` зовёт
`navigate(_curPath, false)` — `navigate()` и есть процедура «вернуть документ в
состояние, которое оставляет свежая загрузка». `push=false`: URL никуда не уходил,
поехал только DOM под ним. Если `navigate()` откажется (устаревшая вьюха успела
инжектнуть инвазивный CSS — см. гейт выше) → `location.reload()`, жёстко.

Рекурсию терминирует `currentViewClass()` (класс, которого хочет `_curPath`): если
устаревший рендер нарисовал ровно его (A → B → A, пока A ещё грузился), то DOM и поллер
корректны и чинить нечего. Он же не даёт двум гоняющимся некэшированным вьюхам запустить
починку починки.

## Hover-prefetch

`wireRouter()` вешает делегированный `pointerover`: при заходе курсора на ссылку
SPA-able-узла — `fetch()` его модуля JS, чтобы прогреть HTTP-кеш браузера
(**не** `require`: тот запустил бы `__init__` и отрендерил чужой view в `#view`).
URL строится `moduleUrl()` байт-в-байт как у `LuCI.require()`
(`base_url/<точки→слэши>.js?v=resource_version`) — иначе кеш-промах. Дедуп по
имени класса (`_prefetched`), ошибки молча глотаются (чистая оптимизация).
Последующий клик → `require()` бьёт по тёплому кешу (−10–40 ms на первый визит
страницы, больше на WAN/VPN). `viewClassFor()` — общий резолвер класса для
navigate() и prefetch.

## Пауза поллинга в фоне

`wireVisibility()` (в `init()`): `visibilitychange` → скрытая вкладка
`L.Poll.stop()` (clearInterval, очередь цела), показ `L.Poll.start()` (пере-арм +
немедленный `step()`). LuCI своего обработчика не имеет, поэтому status/overview в
фоновой вкладке иначе молотит ubus 24/7 (дорогой iwinfo `getAssocList`). Поллер,
добавленный пока скрыто, не авто-стартует (`stop()` удаляет tick) — `start()`
подхватит его на показе, ничего не теряется. Возобновляем только то, что сами и
поставили на паузу (`wasActive`): пользователь мог остановить поллинг вручную
через индикатор «Refreshing», и безусловный `start()` на показе вкладки молча
отменил бы это.

## Проверено (Playwright, живой роутер)

Прогон сделан, когда раскладки ещё были двумя темами с двумя рендерерами; сегодня это
один рендерер и `:root[data-layout]`, поэтому «в обеих раскладках» ниже означает «с
обоими значениями атрибута» — обход всё равно стоит повторять в обоих, чтобы поймать
регрессию в клике по пункту меню, который в баре живёт в дропдауне.

- sidebar и top-nav: system → interfaces → dhcp → system — все переходы SPA
  (маркер `window` жив = нет перезагрузки), контент/подсветка/вкладки/заголовок
  верные, 0 ошибок консоли;
- back/forward через `popstate` — остаётся SPA, восстанавливает нужную страницу;
- сложные CBI-form view (Interfaces, DHCP) рендерятся полностью, save/apply-футер
  на месте, поллеры («Refreshing», аптайм) живые;
- клик на прочие (не-overview) `template`-узлы — корректный полный reload
  (фолбэк). Сам overview с тех пор заведён под SPA через `ensureOverviewHelpers`
  (см. выше) — `viewClassFor` мапит его `template`-узел на `view.status.index`.

**Полный обход (обе раскладки).** Скрипт берёт дерево из `ui.menu.load()`,
перебирает **все 65 кликабельных узлов**, по каждому синтезирует `<a href>` в DOM
и кликает (делегированный листенер роутера ловит его как обычный пункт меню), а
затем — и это главное — делает **настоящую полную загрузку того же URL** и
сравнивает состояния. Сверка с сервером, а не с самим собой: клиентский резолв
alias/firstchild обязан выбрать тот же лист, что и `dispatcher.uc`.

Результат: **62 SPA-OK, 0 расхождений, 3 фолбэка** (openvpn, `admin/vpn`,
logout) — в обеих раскладках. Совпадают `data-page`, `dispatchpath`, `pathinfo`,
URL и число вкладок; консоль чистая. Back/Forward по цепочке из alias- и
firstchild-URL (6 назад + 3 вперёд) — ни одной перезагрузки, `data-page` везде
разрешённый лист.

Ловушка при написании такого обхода: перебирать **листья** дерева нельзя. Узел с
`action` может иметь детей (`admin/status/nftables` — `view` с ребёнком
`iptables`), а именно `alias`/`firstchild` — почти всегда узлы с детьми. Обход по
листьям пропускает ровно те 8 узлов, где и жила ошибка.

Повторить можно playwright-сценарием: пройти по ссылкам меню и проверить, что
маркер, положенный в `window`, переживает переход (жив = перезагрузки не было).

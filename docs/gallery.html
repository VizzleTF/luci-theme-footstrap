Buttons — every cbi-button variant

.btn .btn.primary
.btn.danger .btn.error .btn.success .btn.info .btn.large .btn.small

The colour words (.danger/.error/.success/.info) and size words (.large/.small) come from styles/base, but the theme's plain .btn rule sits in a LATER layer and overrides background/padding/radius — so the theme has to re-state them (theme/55-buttons.css). They rendered as neutral grey buttons, and .btn.large came out with small-button padding, until it did. Nothing in stock LuCI emits them; a third-party luci-app-* can.

Keyboard focus: Tab through this section. Every button must show a ring — there was none at all before (base listed button:hover, not :focus, and the theme's box-shadow: none erased the rest).

What to look at: the create-item row must read as a text field, not as a choice — it is the only row a user can type in. Base gives it width: 100% via li > input.create-item-input:first-child:last-child, which only matches because the input is that li's sole child; the theme paints it as a normal input (theme/65-dropdown.css). Check its focus ring, and check that the row's own hover pill does not fight the field inside it.

The theme also carries a rule for li.create-item, a class stock LuCI does not emit anywhere (grepped: ui.js gives the li no class at all, only the input gets .create-item-input). It is kept under the coverage contract — a third-party app may still emit it — but it is deliberately not rendered here, because this gallery shows the markup LuCI really produces.
